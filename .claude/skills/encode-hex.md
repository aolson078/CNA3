# Encode Hex

Guided hex data entry workflow for digitizing the CNA game map.

## Data Schema

Each hex uses the HexData schema (from `cna/data/maps/hex_schema.py`):

```python
HexData(
    id: str,          # e.g., "C2010"
    col: int,         # column number extracted from id
    row: int,         # row number extracted from id
    terrain_type: str, # see valid types below
    elevation: int,    # 0 = sea level, 1+ = elevation levels
    road: list[str],   # list of adjacent hex IDs connected by road
    track: list[str],  # list of adjacent hex IDs connected by track
    rail: list[str],   # list of adjacent hex IDs connected by rail
    coast: bool,       # True if hex borders the Mediterranean
    features: list[str] # additional features (port, airfield, fortress, etc.)
)
```

### Valid Terrain Types
- `clear_desert` --- open desert, no features
- `rough` --- rough terrain
- `sand_sea` --- impassable sand sea
- `escarpment` --- escarpment hex
- `coastal_plain` --- coastal plain
- `mountain` --- mountain terrain
- `town` --- town hex
- `city` --- major city
- `oasis` --- oasis
- `salt_marsh` --- salt marsh / sebkha
- `wadi` --- wadi

### Valid Features
- `port`, `minor_port`, `airfield`, `fortress`, `supply_dump`, `well`, `pass`

## Procedure

### Step 1 --- Determine entry mode

Ask the user which mode they want:
- **Field-by-field**: Enter one hex at a time, prompted for each field
- **Batch mode**: Enter multiple hexes with shared attributes

### Step 2a --- Field-by-field mode

For each hex:
1. Prompt for hex ID (e.g., `C2010`)
2. Auto-extract col and row from the ID
3. Prompt for terrain_type --- validate against the allowed list
4. Prompt for elevation (default: 0)
5. Prompt for road connections --- list of adjacent hex IDs
6. Prompt for track connections
7. Prompt for rail connections
8. Prompt for coast (y/n)
9. Prompt for features (comma-separated, or empty)
10. **Validate** the entry (see Step 3)
11. Confirm and append to the output file

### Step 2b --- Batch mode

Accept input like:
- `"hexes C2010-C2015: clear desert, elevation 0"`
- `"hexes C2010, C2012, C2014: rough, elevation 1, coast"`
- `"road C2010-C2011-C2012"` (creates road connections between consecutive hexes)
- `"rail C2010-C2011-C2012"` (same for rail)

Parse the batch input, expand ranges, and create individual HexData entries.

### Step 3 --- Validation

For every hex entry, validate:

1. **Hex ID format**: Must match pattern `[A-Z]\d{4}` (letter + 4 digits)
2. **Terrain type**: Must be in the valid terrain types list
3. **Adjacency**: Road, track, and rail connections must reference hex IDs that are geometrically adjacent (on a hex grid, each hex has exactly 6 neighbors)
4. **Reciprocal connections**: If hex A has a road to hex B, warn if hex B doesn't have a road back to hex A (offer to auto-add)
5. **Feature compatibility**: Some features imply constraints:
   - `port` requires `coast: true`
   - `airfield` cannot be in `sand_sea` or `mountain`
6. **Duplicate detection**: Warn if the hex ID already exists in the output file

### Step 4 --- Running count

After each entry or batch:
- Show how many hexes have been entered in total
- If the user has stated an expected total, show progress: `"142 / 2,400 hexes entered (5.9%)"`

### Step 5 --- Output

Write hex data to `cna/data/maps/{map_region}.py` where `map_region` is determined by the hex column letter or asked from the user. Format as a Python module:

```python
"""Map hex data for region {region}.

Auto-generated by /encode-hex. Do not edit manually.
"""

from cna.data.maps.hex_schema import HexData

HEXES: list[HexData] = [
    HexData(id="C2010", col=20, row=10, terrain_type="clear_desert",
            elevation=0, road=["C2011"], track=[], rail=[],
            coast=False, features=[]),
    # ... more hexes
]
```

### Step 6 --- Session persistence

If the user stops and resumes, read the existing output file to pick up where they left off. Report the current count and last hex entered.
